# DIY流程搭便车服务扩展篇

## 1.处理流程拒绝，流程完成等事件。

很多时候我们需要在流程拒绝、流程完成等动作完成以后做一些动作。前面讲到搭便车流程本质上仍然是低代码流程，因此可以使用低代码流程的事件扩展机制。

处理方式也非常简单，我们创建一个类，实现LFFormOperationAdaptor 接口，这个接口定义的流程不同阶段的回调事件，在指定的方法里面处理即可。

> 此接口里面定义了非常多的方法，并非每一个都要实现。不需要的留空即可，关键的方法是finishData，即流程完成方法，流程完成以后会回调它，在里面编写自已的逻辑即可。

> 实现类上使用的是普通的@Service接口，接口的value必须要有，为设计流程的formCode，参照TestLfFormOperationAdaptor

### 常见回调方法

| 方法名              | 描述                                                                                                                                                                                                                                                                                                                                                     |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| PreviewSetCondition | 为流程预览设置参数，表单内容很多，是否全部都需要设置？答案是否定的，如果流程没有使用条件，完全可以不用设置，如果用了条件，只需要设置条件字段的值即可。<br />因为条件字段可能影响流程分支                                                                                                                                                                 |
| launchParameters    | 流程提交时将前端请求参数中和条件有关的参数字段拷贝到BpmnStartConditionsVo对象中，如果当前流程没有条件则不需要设置值，返回空对象即可                                                                                                                                                                                                                      |
| OnInitData          | 留空，不需要实现                                                                                                                                                                                                                                                                                                                                         |
| OnQueryData         | 此方法用于将流程审批时候需要的前端数据从数据库中查出来（一般是存数据库，也可以是其它形式存储，核心是将数据返回，数据怎么来的不重要，也可能是调用接口获取到的<br />，然后填充到方法入参对象里                                                                                                                                                             |
| OnSubmitData        | 流程提交时将用户提交的表单数据入库，这还没完还需要将入库以后数据的Id，给流程自定义个标题和摘要（摘要主要为了后面方法检索流程）以及设置EntityName<br />参数第三方账号申请流程来做。这里之所以要将入库后的数据id返回，因为它要入到bpm_business_process这个表，此表为流程和业务关联的核心表，用于根据流程编号<br />查找到流程实例信息以及流程对应的业务信息 |
| OnConsentData       | 流程同意和重新提交回调方法，如果是重新提交，一般需要更新表单数据。如果审批人可以更改表单，同意事件一般也要更新数据库。                                                                                                                                                                                                                                   |

## 2.解决表单数据不一致问题

上一节的改造方法中我们提到了将表单数据转成HashMap，然后在流程发起时将它传入到构建的对象的lfFields里面。这里其实是将表单数据传给了低代码流程一份，即低代码流程存的是原业务数据的一个副本，原来业务表单可能会发生编辑，流程引擎这边的副本无法实时更新（当然也能做到，只是比较麻烦),这样在流程审批的时候就可能无法获取到最新更新以后的数据。为了解决这个问题，我们需要做一些处理


### 2.1流程提交时必须带上businessId字段，即将业务数据的Id设置到流程提交数据的businessId里，lfFields参数可以不用再设置

> 上一篇中我们实际上使用的是副本数据，要放在lfFields里才能被低代码引擎存入到表里面。这里我们不再需存业务数据了，因此lfFields不需要设置，但是businessId必须要设置。

### 2.2继承AbstractFreeRideFomOperationAdaptor并实现queryBusinessData

我们需要创建一个Service，它继承自抽象类AbstractFreeRideFomOperationAdaptor（它实现了LFFormOperationAdaptor接口）和前面第一节讲的处理流程事件类似，只是那里是实现接口，这里是继承抽象类。必须实现queryBusinessData接口

> 类似于上一篇讲到的流程提交一样，需要将将业务数据映射成HashMap,然后赋值给vo对象lfFields。
>
> 如何获取业务数据呢，2.1里面我们将业务id设置到了businessId字段，流程引擎会根据流程编号把businessId查出来，用户在queryBusinessData根据businessId把相应的业务数据查出来，然后放转成HashMap放在lfFields里面。
