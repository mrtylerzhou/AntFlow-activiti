# AntFow API集成实战

> 关于API集成快速入门也可以查看AntFlow官方快速上手文档.docx 4.3部分三方接入

> 外部接入、三方接入、API接入，SAAS流程实际上指向相同内容。三方接入是以业务接入方为维度来区分的。API接入是从接入的技术形式上来说的。SAAS流程是对外部接入、三方接入概念的升级，强调将流程做为基础服务使用。

## 1.创建接入租户

> 租户是一个管理概念，方便流程引擎系统对外部的接入式进行统一管理和区分。如果是一个公司的多个业务系统接入，那么业务可以做为租户，比如CRM，CMS，WMS系统都可以做为一个独立的租户。如果是以独立的使用主体来区分，则租户就是普通意义上的SAAS租户，通用以公司为单位。

![1772955720286](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772955720286.png)

如上图示，通过antflow管理界面创建一个SAAS租户。

> 调入式完成使用restful api通信。嵌入式即类似钉钉，飞书小程序这样，将外部系统流程做为一个小程序嵌入到当前应用。需要结合企业单点登录系统设计，目前并未实现，如果有企业想要做这样的功能可以联系定制。

调用式和嵌入式不同点在于流程发起方式上。嵌入式实际上页面是外部接入系统设计的，和流程引擎所在系统单点登录打通，然后将流程发起做在流程引擎系统里。而调入式则是三方系统通过restful api来调用流程引擎接口发起一个流程。发起以后流程待办在流程引擎系统里办理。

需要注意的是，调入式在流程引擎系统里面也是有发起入口的。但是实际中往往是业务驱动流程，以wms系统为例，假如有一个领料流程，领料流程往往不是单独发起一个流程（也有可能是），而是在生产过程中某个物料不够自动驱动发起的，这时候后端会有一系统的逻辑处理，然后通过后台调用工作流api来发起一个流程，这是一种比较复杂的业务中穿插有流程发起。此外WMS系统中也会有一些单纯的流程发起，就像在流程系统中发起流程一样，只是流程发起是做在了wms系统里。这时候wms系统后台会处理表单数据，然后调用流程发起api来发起流程）。总之一句话，流程发起往往是在外部系统，并且是通过后台发起的。

当然以上只是笔者的理解，实际中也有可能流程是通过流程引擎系统发起的，外部系统接收发起回调处理表单数据。需要大家多尝试，有无限的可能。

![1772956753413](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772956753413.png)

以上便是通过流程引擎系统发起一个外部流程，但是要处理流程发起回调事件，不然外部系统不知道一个流程发起了。嵌入式长的和上面差不多，区别在于以上是低代码设计的，嵌入式页面往往是外部系统自己设计的，通过打通单点登录接入到流程引擎系统。

## 2.创建应用

> 一个租户下面可以创建多个应用，一个应用即一个流程。

![1772957077862](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772957077862.png)

> 目前应用类型无法选择。类型为应用和为流程目前无任何区别。父级应用是一个管理容器，在流程发起页面会出现一个球，点开球是一系列小球，方便同一个业务方多个业务系统的管理。前面说了，目前不支持嵌入式，而调入式的发起往往是三方系统由后台通过httpclient发起的，因此父应用管理概念目前阶段意义不大，不支持选择。

创建完以后，在应用管理tab里就能看到它了

## 3.设置回调

应用管理右侧操作栏有个**齿轮**图标，点击它便可以设计流程回调了。

![1772957577020](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772957577020.png)

> 有些用户可能之前没接触过API接入流程，不知道回调是干嘛的。前面说了，api接入流程发起一般是由外部系统通过调用api发起的，流程流转（同意，拒绝，完成）是在流程引擎系统中完成的，如果没有回调，外部系统并不知道流程发生了什么操作。因此需要通过回调来将流程的流转告诉外部系统。

> 注意页面上没有校验回调添加，实际上只能添加一个回调url，此时有些用户可能会有疑问。流程流转事件可能会有很多，为什么只有一个url呢，实际上流转事件是通过回调的参数里的eventType（值查看枚举CallbackTypeEnum）来确定的，而不是通过不同的url来区分。用户使用一个接口接收全部回调事件，通过区分eventType来确定当前是会么回调事件。

## 4.设置审批人选择api

上面应用管理设置页面还有一个设置审批人按钮，点击便可以设计审批人获取接口

![1772958321100](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772958321100.png)

> 需要注意的是，对于同一个公司有多套系统，共享同一套用户基础信息的以上非必须，但是外部流程前端校验必须要设计审批人员接口，因此这一步不可避免。对于完全SAAS化的系统（即不同租户之间数据完全隔离，没有任何关系），以上是必须的。对于完全SAAS化流程，如果不提供人员选择接口，就没法设计流程节点审批人。对于共享同一套用户系统的SAAS流程，将人员获取接口写在流程引擎系统里就可以了。

> 需要注意的是审批人回调url前端是灰色的，防止用户随便输入导致流程体验报错，用户下载到本地以后可以将限制放开。还是那句话，对于非完全SAAS化的流程，这个设计是非必要的，用户随便设计一条即可。

流程发布完以后，页面会跳到【流程设计】tab下的列表里，在这里还要点击【启动】来使刚刚设计的流程生效。

![1772959032015](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772959032015.png)

## 6.发起流程

流程设计完成并启动以后，【流程中心】下面的【第三方流程】里便会出现一个球，点击球即可发起一个外部流程。

> 如果没有看到，刷新一下页面

![1772959262313](image/AntFlow业务集成之三API接入(SAAS)流程实战/1772959262313.png)

前面说过外部流程发起往往不是这样发起的，外部api接入流程往往发起是在外部系统，外部系统处理完表单以后通过后台调用流程引擎系统api来发起流程，发起完以后拿到流程编号，然后将流程编号和业务数据进行绑定。当然也是可以像上面介绍的一样直接通过点击第三方流程的球来发起外部流程，外部系统接收回调事件来处理流程发起后的操作。

> 通过外部api来调用流程发起操作需要知道流程发起的接口以及请求的数据结构，我们点击刚创建的球同时打开f12，便可以看到请求的地球和请求参数了

### 6.1 外部api接入请求地址:POST /outSide/processSubmit

### 6.2外部api请求参数

请求参数每个流程每次请求都不一样，但是格式是固定的，下面我们就刚创建的爱德博格大学入学流程来分析一下

```json
{
	"formCode": "EDBERG_ADBGDXRXLC",
	"operationType": 1,
	"isLowCodeFlow": true,
	"lfFields": {
		"input28420": "张小华",
		"input18959": "24"
	},
	"userId": "1"
}
```

#### 6.2.1 核心参数解析

+ formCode formCode是antflow里面非常重要的一个概念，从字面意义理解为表单编号,实际上它不仅仅是表单，而且是一个流程的惟一标识。即我们创建的每一个流程都有一个惟一的formCode,流程编号的前缀就是formCode,我们对维护的流程熟悉了，通过流程编号就可以知道当前流程是一个什么样的流程。

> 通过上面f12我们通过抓取请求参数可以找到某个流程的formCode，还有其它手段。

1. 通过流程和表单业务关联表找到它。select * from bpm_business_process where BUSINESS_NUMBER='流程编号'; 这种方法的前提是流程已经发起了，这就涉及到了先有鸡还是先有蛋的问题。我们发起流程之前就要知道formCode，更一般的是通过流程设计表找到它。即查询bpmn_conf表。思路是我们设计流程的时候已经知道它的名字了，我们根据名字来查formCode,select * from t_bpmn_conf where bpmn_name='爱德博格大学入学流程'; 需要注意的是以上查询会查出多条记录,是流程的不同版本，但是formCode是一样的。effective_status值为1的是当前活跃的版本，即正在使用的版本。

题外话：很多用户想要删除流程历史上的版本，实际上以上语句即可查找出所有的流程版本，我们可以选择性删除，也可以一次删除所有非活跃的。


* operationType,流程提交值固定为1，antflow流程的发起、同意、拒绝等操作都是通过同一个接口来完成的，使用的是策略模式，通过operationType找到不同的处理策略。都有哪些值可以查看ProcessOperationEnum枚举。
* isLowCodeFlow值是固定的，为true。目前模式为低代码模式。（外部流程也可以使用自定义表单形式，但是比较复杂，有需要的用户可以沟通定制）
* lfFields即表单参数，外部流程的表单数据都要放在lfFields里面。如果用户在设计表单的时候不对表单字段进行更改，默认就像上面一样是inputxxxxx这样的很不方便记忆。设计的时候一般要与三方系统里面的表单参数命名一样。这样在提交流程的时候就好处理了，可以遍历所有表单字段放在一个HashMap里（.net是字典)。
* userId 发起用户的Id，这个参数是必须的，即流程是由谁发起的，由于是通过api发起的，流程引擎系统无法获取当前用户，只能通过外部系统告诉引擎系统当前发起人是谁。

### 6.2.2困惑及说明

有些用户看完以后可能会有一个很大的困惑。前面讲到过，api流程往往是通过三方系统发起的，并且往往是通过后台httpclient调用api发起的。表单是在外部系统的，这里设计的时候为什么还需要设计表单呢？之所以要设计表单，是因为接入模式决定的，前面讲到过，这种模式流程发起是由外部系统调用api发起的，流程流转（同意，拒色，打回，终止等）操作都是在流程引擎系统。流程审批需要有表单数据，如果没有表单审批人不知道当前流程是干嘛的，自然也就没法确定是要同意还是拒绝。因此需要设计表单，这里流程引擎存的表单数据实际上是外部系统业务数据的一个副本。主要展示给审批人看的。基于以上，我们可以得出以下一个结论。这里设计的表单字段不是必须和在外部系统里的内容完全1:1,流程审批时审批人往往只需要看到一些核心的数据便可以做出决策，因此用户不需要完全复刻在三方系统的表单内容。只需要关键信息即可。
