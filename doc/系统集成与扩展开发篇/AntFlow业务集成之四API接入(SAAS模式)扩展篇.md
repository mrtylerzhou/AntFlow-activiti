# 外部接入模式扩展篇

前面我们介绍了SAAS流程的使用场景以及最小化配置接入。实际上上一篇的接入是一种最小化的接入，并不是外部接入的全貌。这里我们再介绍一些更丰富的内容，以抛砖引玉，帮助大家打开思路，实现更多更丰富的功能。

> 前面一篇讲的内容是外部系统通过API进行流程提交，然后在流程引擎系统里面进行审批。实际上还可以将所有功能都对接到外部系统里，仅将流程引擎系统当做流转工具。

## 1.实现流程预览:

上一篇我们讲到了如何通过API调用来提交一个外部流程，和内部流程对比，我们发现内部流程在发起时都有一个预览功能。我们在外部系统发起一个流程时如何像在流程引擎系统里一样，发起之前预览一下呢？其实也非常简单，前面我们讲到过，虽然外部流程往往由外部系统通过api发起，但是它在流程引擎系统里面也有发起入口，我们点击它的球，球里面是有预览功能的，我们通过f12抓取请求路径和请求参数，在外部系统里按照同样的格式拼接请求参数调用相同的预览接口即可。

> 需要注意的是预览接口并没有userId字段，我们需要在header里面添加userId请求头，然后带上当前登录用户的id。后面其它的接口也都是一样。

## 2.对接我的发起、待办、已办列表。

同样是通过f12抓取流程系统里相对接页面的请求api，找到请求参数和响应参数，然后设计对应的列表。

> 需要注意的是，在流程系统里，我的待办，我的已办都是全部流程数据，做为SAAS系统的某个业务系统只需要自己的数据，根据响应条目数据的processKey（即为formCode）来过滤本SAAS租户的数据。

> 待办、已办这些都是列表，用户可以直接将流程引擎系统里的列表页面复制过去，也可以自己重新设计，重新设计的只需要api调用相同即可。风格和展示字段都可以灵活定制。

> 租户在自己系统对接了我的待办、已办等列表数据以后，这些数据在流程引擎系统里面仍然是有的。用户仍然可以在流程引擎系统里办理，如果不想让它在流程引擎系统里办理，需要在流程引擎系统里过滤掉特定租户的数据。

通过以上api的对接，流程引擎系统的功能只有租户管理，流程设计这些。其它的都在业务系统里，这样体验会好一些，但是对接的工作量增加不少，用户要结合实际场景来评估。需要说明的是，本篇里的内容都是扩展对接，并不是必须的。实际上最小化对接和上一篇讲的一样，只需要对接一个发起api，然后流转都在流程引擎系统。
